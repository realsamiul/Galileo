


# WhatsApp vs Messenger: The Setup Decision

## The Honest Comparison

| Factor | WhatsApp Business API | Messenger (Meta Graph API) |
|---|---|---|
| **Where your users are** | Everyone in Bangladesh. Period. | Facebook users — same people, but they don't message businesses there first |
| **AWS native integration** | AWS End User Messaging Social (managed, current) | No AWS-managed equivalent — you'd build a webhook yourself |
| **Verification timeline** | WhatsApp Business Account approval: 1-7 days typical, can take weeks if flagged | Instant if you have a Facebook Page + App |
| **Cost per conversation** | Meta charges per 24-hour conversation window (~$0.03-0.08 for Bangladesh, service conversations) | Free (no per-message fee from Meta) |
| **Message types** | Text, voice notes, images, documents (PDF), location | Text, images, files (less clean for document return) |
| **Voice notes** | Native, extremely common in BD | Exists but rarely used for business |
| **Document sending** | Native PDF/DOCX attachment in chat | File attachment works but less elegant |
| **User trust for business** | High — people already message businesses on WhatsApp in BD | Medium — feels more casual/personal |
| **Setup complexity** | Medium-High (WABA registration, phone number, verification) | Medium (Facebook App, Page, webhook) |

## My Recommendation

**Start with WhatsApp.** Here's why:

1. Proof Sheet's killer use case is voice note → PDF back in the same chat. WhatsApp voice notes are *the* communication mode in Bangladesh. Messenger voice exists but isn't habitual for business.

2. Returning a PDF as a WhatsApp document attachment is clean — it appears inline, downloadable, shareable. This is the product experience.

3. AWS End User Messaging Social exists specifically to manage this. It handles the Meta Cloud API complexity for you.

4. Your target user (shop owner, student, freelancer in BD) already has business conversations happening on WhatsApp. You're inserting into existing behavior, not creating new behavior.

**However** — there's a pragmatic sequencing option I'll flag at the end.

### Step-by-Step: Meta Side (Do This First)

**Step 1: Meta Business Account**

Go to `business.facebook.com`. If you already have one, confirm you see your business name at the top left. If not, create one — takes 5 minutes, needs your business name and your name.

Business verification (the blue tick process) is NOT required to start. You can send up to 250 unique conversations/day without it. Verify later when volume justifies it.

**Step 2: Create a Meta App**

Go to `developers.facebook.com/apps`

- Click "Create App"
- Select "Other" for use case (or "Business" if prompted)
- App type: "Business"
- Name it something like "ProofSheet" or your agency name
- Link it to your Meta Business Account when prompted

**Step 3: Add WhatsApp Product**

Inside your new app's dashboard:
- Left sidebar → "Add Products"
- Find "WhatsApp" → click "Set Up"
- This creates a WhatsApp Business Account (WABA) linked to your Meta Business Account
- Meta gives you a **test phone number** and **test recipient** for free — you can build your entire pipeline against this before registering your real number

**This is important:** You do NOT need your own phone number yet to start building. Meta provides a test sender number. You can only send to numbers you add as test recipients (up to 5), but that's enough to build and verify the entire pipeline.

**Step 4: Get Your Credentials**

From the WhatsApp section of your app dashboard, you need:
- **Phone Number ID** (the test number's ID)
- **WhatsApp Business Account ID**
- **Temporary Access Token** (valid 24 hours — fine for building)
- Later: a **Permanent System User Token** (for production)

**Step 5: Configure a Webhook**

This is where AWS enters. Meta needs a URL to send incoming messages to. You'll provide an API Gateway URL (set up in the AWS steps below). For now, just know that in the WhatsApp settings there's a "Configuration" section where you paste a webhook URL and a verify token.

### Step-by-Step: AWS Side

#### Option A: AWS End User Messaging Social (Managed)

This is the "proper" path from your technical briefing. AWS manages the Meta connection for you.

**Current status:** This service handles the webhook management, message receipt, and message sending through a managed interface. You configure your WABA in the AWS console, and messages arrive via SNS topics.

Setup via AWS Console:
1. Go to AWS End User Messaging (in the console, search "End User Messaging")
2. Navigate to Social messaging
3. Link your WhatsApp Business Account (requires the WABA ID from Meta)
4. AWS creates an SNS topic that receives all inbound messages
5. You subscribe a Lambda function to that topic

**Advantage:** AWS handles webhook verification, token management, retry logic.
**Disadvantage:** Slightly less control over the raw message format. Newer service, fewer Stack Overflow answers if you hit edge cases.

#### Option B: Self-Managed Webhook (More Control, Also Fine)

This is the path most developers actually take and what the `build-on-aws` sample repository uses:

```
Meta WhatsApp Cloud API 
    → sends webhook POST to your API Gateway URL
    → API Gateway triggers Lambda
    → Lambda processes message
    → Lambda calls WhatsApp API to reply
```

Setup:
1. Create a Lambda function (Python 3.12 runtime)
2. Create an API Gateway (HTTP API — simpler than REST API)
3. Connect API Gateway → Lambda
4. Take the API Gateway URL, paste it into Meta's webhook configuration
5. Handle Meta's verification challenge (they send a GET with a token, you echo it back)

**I recommend Option B for you right now.** Here's why:
- The `building-gen-ai-whatsapp-assistant-with-amazon-bedrock-and-python` sample repo uses this exact pattern — you have reference code
- You have full visibility into every message as raw JSON
- Debugging is simpler (CloudWatch logs show you exactly what Meta sent)
- You can add SQS between API Gateway and the processing Lambda later for throughput (the architecture in the briefing) without changing the Meta-facing endpoint

Here's the skeleton. This is the single function that receives a WhatsApp message and replies:

import json
import boto3
import urllib3

# Clients
bedrock = boto3.client('bedrock-runtime', region_name='us-east-1')
secrets = boto3.client('secretsmanager')

WHATSAPP_TOKEN = "your-token"  # Move to Secrets Manager for prod
PHONE_NUMBER_ID = "your-phone-number-id"

def lambda_handler(event, context):
    body = json.loads(event['body'])
    
    # Meta sends various webhook events — filter for actual messages
    if body.get('object') != 'whatsapp_business_account':
        return {'statusCode': 200}
    
    entries = body.get('entry', [])
    for entry in entries:
        changes = entry.get('changes', [])
        for change in changes:
            value = change.get('value', {})
            messages = value.get('messages', [])
            
            for message in messages:
                sender = message['from']  # User's phone number
                msg_type = message['type']  # text, audio, image, document
                
                if msg_type == 'text':
                    user_text = message['text']['body']
                elif msg_type == 'audio':
                    # Handle voice note — Phase 2
                    audio_id = message['audio']['id']
                    user_text = transcribe_voice(audio_id)
                else:
                    user_text = "[unsupported message type]"
                
                # Generate response via Bedrock
                response = generate_response(sender, user_text)
                
                # Send reply
                send_whatsapp_message(sender, response)
    
    return {'statusCode': 200}


def generate_response(sender, user_text):
    """Call Bedrock Claude with the user's message"""
    
    response = bedrock.invoke_model(
        modelId='anthropic.claude-sonnet-4-20250514',
        body=json.dumps({
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 1024,
            "system": SYSTEM_PROMPT,
            "messages": [
                {"role": "user", "content": user_text}
            ]
        })
    )
    
    result = json.loads(response['body'].read())
    return result['content'][0]['text']


def send_whatsapp_message(to, text):
    """Send a text reply via WhatsApp Cloud API"""
    
    http = urllib3.PoolManager()
    url = f"https://graph.facebook.com/v21.0/{PHONE_NUMBER_ID}/messages"
    
    payload = {
        "messaging_product": "whatsapp",
        "to": to,
        "type": "text",
        "text": {"body": text}
    }
    
    http.request(
        'POST', url,
        body=json.dumps(payload),
        headers={
            'Content-Type': 'application/json',
            'Authorization': f'Bearer {WHATSAPP_TOKEN}'
        }
    )

available on WhatsApp. You help users create professional 
documents by asking for the needed information conversationally.

You handle: invoices, receipts, business letters, leave 
applications, CVs, and more.

- Handle Bangla, English, and code-mixed naturally

Example output when ready:
[DOCUMENT_READY]

### The Messenger Fallback Option

If WhatsApp Business verification becomes slow or problematic, Messenger is your instant fallback:

- Same Meta App, just add the "Messenger" product instead
- Webhook structure is nearly identical (different JSON schema but same concept)
- No phone number needed — messages come to your Facebook Page
- Free (no per-conversation charge)
- Works immediately with no verification wait

You could genuinely build the entire pipeline against Messenger first (since it's instant), then swap the channel layer to WhatsApp once WABA is approved. # WhatsApp Setup: Troubleshooting Guide

## Common Blocks You'll Hit (And How to Clear Them)

### Block 1: "You need a Meta Business Account"

If `developers.facebook.com` tells you that you need a Meta Business Account to create a Business type app:

- Go to `business.facebook.com/overview` first
- Create one there (business name, your name, your email)
- Then return to developers.facebook.com and try again
- The link between the two isn't always instant — give it 5 minutes, refresh

### Block 2: WhatsApp Product Doesn't Appear in "Add Products"

This occasionally happens if your Meta Business Account is very new or flagged for review.

- Clear fix: make sure your app type is "Business" not "Consumer" or "Gaming"
- If still missing: go to `developers.facebook.com/apps/YOUR_APP_ID/whatsapp-business/wa-dev-console` directly — sometimes the product is accessible via URL even when the button doesn't render
- If truly blocked: your Meta Business Account may need email verification. Check Settings → Business Info on business.facebook.com

### Block 3: "Two-Factor Authentication Required"

Meta enforces 2FA for developer accounts in most regions now. If prompted:

- Set it up immediately (authentication app, not SMS if possible)
- Don't skip this — you'll be locked out of the webhook configuration later if you defer

### Block 4: Test Number Setup Confusion

When you add WhatsApp to your app, Meta gives you:

- A **Meta test phone number** (something like +1 555 XXX XXXX) — this is the *bot's* number, the one that sends replies
- You then add **your own real WhatsApp number** as a "To" recipient for testing
- You must add your number in the API Setup panel under "To" field and send yourself a test template message first to confirm the connection

**The confusing part:** you cannot just message the test number from your WhatsApp. You must first *send* a template message from the API/dashboard *to* your number, and *then* you can reply to it and trigger webhooks. This is Meta's 24-hour conversation window rule — the bot must initiate with an approved template, then freeform conversation opens for 24 hours.

**For testing however:** Meta's test number has a pre-approved "hello_world" template. In the API Setup panel, there's a "Send Message" button that sends this template to your test recipient number. Click it. You'll get a WhatsApp message. Now you can reply to it and your webhook will receive that reply.

### Block 5: Phone Number Verification Loop

If you proceed to register your own number (not needed immediately — test number works first):

- Meta sends a 6-digit code via SMS or voice call to verify ownership
- If SMS doesn't arrive: switch to voice call option
- If neither arrives: the number may still be registered on WhatsApp/WhatsApp Business app. You **must** delete the account from the app first (Settings → Account → Delete my account), wait 5 minutes, then retry verification
- Grameenphone and Robi numbers in Bangladesh work fine. Banglalink occasionally has SMS delivery delays for short codes — voice call is more reliable

### Block 6: Webhook Configuration Won't Verify

This one you'll hit when you try to paste your API Gateway URL into Meta's webhook field. Meta sends a GET request with a challenge string and expects it echoed back. Common failures:

- Your Lambda returns JSON instead of plain text — the challenge must be returned as raw text with status 200
- Your API Gateway has CORS configured wrong — for webhook verification, it doesn't matter, but make sure you're not requiring an API key
- Meta times out — your Lambda cold start took too long. If you're using Python with no layers, cold start is <1 second, you should be fine
- **The verify token you entered in Meta must exactly match what your Lambda checks for** — copy-paste, don't retype

### Block 7: "Business Verification Required" for Higher Messaging Limits

You start with 250 unique conversations/day. That's fine for building and early users. Don't worry about business verification until you have traction. When you do need it:

- Meta asks for a business document (trade license, utility bill with business name, or bank statement)
- Bangladeshi trade licenses work. Upload a clear photo/scan.
- Verification takes 2-10 business days
- You can build and test everything in the 250/day limit

### Block 8: Temporary Token Expires

The token Meta gives you in the API Setup panel lasts 24 hours. For building over multiple days:

- Option 1: Generate a new temporary token each session (fine for development)
- Option 2: Create a System User in Business Settings → Users → System Users → generate a permanent token with `whatsapp_business_messaging` permission. Do this before going to production.

### The Lambda Code (Full — Deploy This)

Create a Lambda function:
- Runtime: Python 3.12
- Architecture: arm64 (cheaper)
- Memory: 512 MB
- Timeout: 30 seconds
- Role: the one you created above

**lambda_function.py:**

```python

import boto3
import urllib3
import hashlib
from datetime import datetime, timezone

# --- AWS Clients ---
bedrock = boto3.client('bedrock-runtime', region_name='us-east-1')
dynamodb = boto3.resource('dynamodb')
secrets_client = boto3.client('secretsmanager')
s3 = boto3.client('s3')

# --- Config ---

DOCS_BUCKET = 'proofsheet-docs-REPLACE-WITH-YOURS'
http = urllib3.PoolManager()

# --- Load secrets (cached across warm invocations) ---
_secrets_cache = None

def get_secrets():
    global _secrets_cache
    if _secrets_cache is None:
        resp = secrets_client.get_secret_value(SecretId='proofsheet/whatsapp')
        _secrets_cache = json.loads(resp['SecretString'])
    return _secrets_cache


# --- Main Handler ---
def lambda_handler(event, context):
    method = event.get('requestContext', {}).get('http', {}).get('method', 'POST')
    
    # Webhook verification (GET)
    if method == 'GET':
        return handle_verification(event)
    
    # Message processing (POST)
    try:
        body = json.loads(event.get('body', '{}'))
    except json.JSONDecodeError:
        return {'statusCode': 200}
    
    if body.get('object') != 'whatsapp_business_account':
        return {'statusCode': 200}
    
    for entry in body.get('entry', []):
        for change in entry.get('changes', []):
            value = change.get('value', {})
            
            # Skip status updates (delivered, read receipts)
            if 'messages' not in value:
                continue
            
            for message in value['messages']:
                process_message(message)
    
    # Always return 200 quickly — Meta will retry on non-200
    return {'statusCode': 200}


def handle_verification(event):
    params = event.get('queryStringParameters', {})
    mode = params.get('hub.mode', '')
    token = params.get('hub.verify_token', '')
    challenge = params.get('hub.challenge', '')
    
    secrets = get_secrets()
    
    if mode == 'subscribe' and token == secrets['verify_token']:
        return {
            'statusCode': 200,
            'headers': {'Content-Type': 'text/plain'},
            'body': challenge
        }
    
    return {'statusCode': 403, 'body': 'Forbidden'}


# --- Message Processing ---
def process_message(message):
    sender = message['from']
    msg_id = message['id']
    msg_type = message['type']
    timestamp = datetime.now(timezone.utc).isoformat()
    
    # Extract text based on message type
    if msg_type == 'text':
        user_text = message['text']['body']
    elif msg_type == 'interactive':
        # Button/list replies
        interactive = message.get('interactive', {})
        if 'button_reply' in interactive:
            user_text = interactive['button_reply']['title']
        elif 'list_reply' in interactive:
            user_text = interactive['list_reply']['title']
        else:
            user_text = '[interactive message]'
    else:
        # For now, handle only text. Voice/image comes in Phase 2
        send_reply(sender, "এখন শুধু text message support করি। "
                          "Voice note আর image support শীঘ্রই আসছে! 🙏")
        return
    
    # Load conversation history
    history = get_conversation_history(sender)
    
    # Generate response via Bedrock
    assistant_response = call_bedrock(user_text, history)
    
    # Check if response contains a document-ready signal
    if '[DOCUMENT_READY]' in assistant_response:
        handle_document_generation(sender, assistant_response)
    else:
        send_reply(sender, assistant_response)
    
    # Save to conversation history
    save_message(sender, timestamp, 'user', user_text)
    save_message(sender, timestamp + '_reply', 'assistant', assistant_response)

def call_bedrock(user_text, history):
    # Build messages array from history
    messages = []
    for msg in history[-10:]:  # Last 10 messages for context
        messages.append({
            "role": msg['role'],
            "content": msg['content']
        })
    messages.append({"role": "user", "content": user_text})
    
    response = bedrock.invoke_model(
        modelId='anthropic.claude-sonnet-4-20250514',
        body=json.dumps({
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 2048,
            "system": SYSTEM_PROMPT,
            "messages": messages
        })
    )
    
    result = json.loads(response['body'].read())
    return result['content'][0]['text']


# --- Document Generation ---
def handle_document_generation(sender, response):
    # Split response into conversational part and JSON
    parts = response.split('[DOCUMENT_READY]')
    conversational_reply = parts[0].strip()
    
    # Extract JSON from the document part
    json_part = parts[1].strip()
    # Remove markdown code fences if present
    json_part = json_part.replace('```json', '').replace('```', '').strip()
    
    try:
        doc_data = json.loads(json_part)
    except json.JSONDecodeError:
        send_reply(sender, conversational_reply + 
                   "\n\n⚠️ Document generation এ একটু সমস্যা হয়েছে। "
                   "আবার try করুন।")
        return
    
    # Generate HTML from template
    doc_type = doc_data['type']
    html_content = render_document(doc_type, doc_data['data'])
    
    # Convert to PDF
    # For Phase 1 without WeasyPrint layer: generate HTML and send as document
    # Full PDF generation covered in the renderer section below
    
    # For now: send the conversational reply
    send_reply(sender, conversational_reply + 
               "\n\n📄 আপনার document তৈরি হচ্ছে...")
    
    # Generate and send PDF
    pdf_bytes = generate_pdf(html_content)
    
    if pdf_bytes:
        # Upload to S3
        doc_key = f"{sender}/{doc_type}_{datetime.now(timezone.utc).strftime('%Y%m%d_%H%M%S')}.pdf"
        s3.put_object(
            Bucket=DOCS_BUCKET,
            Key=doc_key,
            Body=pdf_bytes,
            ContentType='application/pdf'
        )
        
        # Send document via WhatsApp
        send_document(sender, pdf_bytes, f"{doc_type}.pdf")
    else:
        send_reply(sender, "⚠️ PDF generation এ সমস্যা হয়েছে। "
                          "আবার চেষ্টা করুন।")

# --- Conversation Memory ---
def get_conversation_history(phone_number):
    """Get last N messages for this user"""
    try:
        response = CONVERSATION_TABLE.query(
            KeyConditionExpression=boto3.dynamodb.conditions.Key('phone_number').eq(phone_number),
            ScanIndexForward=False,  # newest first
            Limit=20
        )
        items = response.get('Items', [])
        # Reverse to chronological order
        items.reverse()
        return [{'role': item['role'], 'content': item['content']} for item in items]
    except Exception:
        return []


def save_message(phone_number, timestamp, role, content):
    """Save a message to conversation history"""
    try:
        CONVERSATION_TABLE.put_item(Item={
            'phone_number': phone_number,
            'timestamp': timestamp,
            'role': role,
            'content': content[:5000]  # Truncate very long messages
        })
    except Exception:
        pass  # Non-critical — don't fail the response


# --- WhatsApp API ---
def send_reply(to, text):
    """Send a text message via WhatsApp Cloud API"""
    secrets = get_secrets()
    url = f"https://graph.facebook.com/v21.0/{secrets['phone_number_id']}/messages"
    
    # WhatsApp has a 4096 character limit per message
    if len(text) > 4000:
        # Split into chunks
        chunks = [text[i:i+4000] for i in range(0, len(text), 4000)]
        for chunk in chunks:
            _send_text(url, secrets['access_token'], to, chunk)
    else:
        _send_text(url, secrets['access_token'], to, text)


def _send_text(url, token, to, text):
    payload = {
        "messaging_product": "whatsapp",
        "recipient_type": "individual",
        "to": to,
        "type": "text",
        "text": {"preview_url": False, "body": text}
    }
    
    http.request(
        'POST', url,
        body=json.dumps(payload),
        headers={
            'Content-Type': 'application/json',
            'Authorization': f'Bearer {token}'
        }
    )


def send_document(to, file_bytes, filename):
    """
    Send a document (PDF/HTML) via WhatsApp Cloud API.
    
    WhatsApp requires documents to be uploaded to their media 
    endpoint first, then referenced by media_id in the message.
    """
    secrets = get_secrets()
    
    # Step 1: Upload media
    upload_url = f"https://graph.facebook.com/v21.0/{secrets['phone_number_id']}/media"
    
    # multipart/form-data upload
    # Using urllib3's request with fields
    resp = http.request(
        'POST', upload_url,
        fields={
            'file': (filename, file_bytes, 'application/pdf'),
            'messaging_product': 'whatsapp',
            'type': 'application/pdf'
        },
        headers={
            'Authorization': f'Bearer {secrets["access_token"]}'
        }
    )
    
    upload_result = json.loads(resp.data)
    media_id = upload_result.get('id')
    
    if not media_id:
        # Fallback: send as text
        send_reply(to, "⚠️ Document upload failed. Please try again.")
        return
    
    # Step 2: Send document message
    msg_url = f"https://graph.facebook.com/v21.0/{secrets['phone_number_id']}/messages"
    payload = {
        "messaging_product": "whatsapp",
        "recipient_type": "individual",
        "to": to,
        "type": "document",
        "document": {
            "id": media_id,
            "filename": filename
        }
    }
    
    http.request(
        'POST', msg_url,
        body=json.dumps(payload),
        headers={
            'Content-Type': 'application/json',
            'Authorization': f'Bearer {secrets["access_token"]}'
        }
    )

Send the HTML file directly as a WhatsApp document with a `.html` extension. When the user opens it on their phone, it renders in the browser — looks perfect. Not ideal (users expect PDF) but lets you ship and get feedback immediately while you set up proper PDF generation.

Change `send_document` call to use `text/html` and filename `invoice.html`.

### Option 2: Lambda Layer with wkhtmltopdf (Ship in 2-3 Days)

There's a well-maintained Lambda Layer for `wkhtmltopdf`:

- Layer ARN (check for your region): `arn:aws:lambda:us-east-1:764866452798:layer:chrome-aws-lambda:xx`
- Or build your own layer with the `wkhtmltopdf` binary for Amazon Linux 2023

The PDF generation function becomes:

```python
import subprocess
import tempfile

def generate_pdf(html_content):
    """Convert HTML to PDF using wkhtmltopdf in Lambda layer"""
    with tempfile.NamedTemporaryFile(suffix='.html', delete=False) as html_file:
        html_file.write(html_content.encode('utf-8'))
        html_path = html_file.name
    
    pdf_path = html_path.replace('.html', '.pdf')
    
    subprocess.run([
        '/opt/bin/wkhtmltopdf',
        '--page-size', 'A4',
        '--margin-top', '10mm',
        '--margin-bottom', '10mm',
        '--margin-left', '10mm',
        '--margin-right', '10mm',
        '--encoding', 'UTF-8',
        html_path,
        pdf_path
    ], check=True, capture_output=True)
    
    with open(pdf_path, 'rb') as f:
        return f.read()
```

### Option 3: Separate Renderer Lambda with Container Image (Most Robust)

Package WeasyPrint (better CSS support, better typography) in a Lambda container image:

```dockerfile
FROM public.ecr.aws/lambda/python:3.12

RUN yum install -y pango cairo gdk-pixbuf2 libffi-devel
RUN pip install weasyprint

COPY lambda_renderer.py ${LAMBDA_TASK_ROOT}/

```

This is the production path. WeasyPrint handles:
- Proper fonts (including Bengali fonts — install Noto Sans Bengali in the container)
- CSS flexbox/grid properly
- Page breaks correctly
- Print-quality PDF output

**My recommendation:** Ship with Option 1 (HTML) today, move to Option 2 (wkhtmltopdf layer) within the first week, and move to Option 3 (WeasyPrint container) when you have paying users and want Bengali-text PDFs to look perfect.
